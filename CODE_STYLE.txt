Правила кодирования в проекте:

1) Не должно быть закоментированных участков кода, если вы видите закоментированный код, его нужно удалить
2) Не должно быть не используемых методов, классов или переменных, их следует удалять
3) В представлениях не должно быть бизнес логики, бизнес логику следует помещать в контроллеры или в модели
4) Контроллеры не должны обращаться напрямую к методам других контроллеров, коммуникация между контроллерами осуществляется с помощью сообщений
5) Размер методов не должен превышать 50 строк, рекомендованная длина 15-20 строк
7) Весь код (в каталоге app) следует проверять утилитой jshint (с настройками описанными в файле .jshintrc ), код вызывающий предупреждения должен исправляться,
    (изменения в конфиге jshint.json должны обсуждаться как минимум с еще одним членом команды)
8) Не допускается смешивание различных стилей обращения к DOM объектам, обращение к DOM объектам осуществляется с помощью методов фреймворка ExtJS
9) Следует избегать использования конструкции для циклов for (;;) или for(var o in items) в пользу метода Ext.each
10) Все сообщения отображаемые для пользователя должны быть объявлены как свойства класса и позволять переопределение с помощью конструкции override
11) Использование глобальных переменных ЗАПРЕЩЕНО
12) Глобальные настройки приложения следует объявлять при вызове Ext.application
13) В каталоге app/views запрещено использовать функции,
    все обработчики событий должны быть размещены в соответствующих контролллерах,
    для переопределения поведения компонентов нужно использовать каталог app\overrides
14) Для именования методов, переменных, членов классов используется нотация camelCase (первая буква в нижнем регистре);
    исключением могут являться названия тестов, т.к. название может быть длинным, слова можно разделять пробелами или символом _.
    Имена классов должны начинаться с Заглавной буквы
15) Общий принцип написания кода: нужно стремиться к компактности кода, но не в ущерб его читаемости

Правила работы с элементами представлений:
1) Родитель может обращаться напрямую к дочерним элементам.
2) Дочерние элементы сообщают информацию «наверх» только через механизм событий.
    Дочерний элемент не должен делать никаких предположений о родителе и наборе его методов.
3) «Горизонтальные» связи между дочерними элементами (вызовы методы друг друга напрямую) строго запрещены.
    Только родитель занимается синхронизацией «дочек».

Правила написания модульных тестов

1) Название имя файла: test + 'Название тестируемого модуля'
2) Тестовый метод должен быть настролько короток и прост насколько это возможно
3) В каждом тестовом методе должно быть возможноым выделить три стадии:
    подготовка тестируемого окружения (Arrange), выполнение тестируемого метода (Act), проверка результата (Assert)
    Такая организация тестов описывается паттерном AAA
4) Если объем тестовых данных слишком велик, его следует выносить в отдельный файл (фикстуру) и динамически подгружать
5) Если есть множество тестов, которые отличаются лишь входными данными не следует использовать цикл для устранения дублирования,
    в этом случае следует использовать конструкцию DataProvider (для этого нужно подключить скрипт /tests/jstestdriver/TestCase.js)
